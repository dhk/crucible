## Summary

The adversarial pass hardened the document's failure modes but in doing so narrowed its conceptual aperture — it now has more dead ends than live branches in several sections. This strawman pass adds five underdeveloped areas: a temporal framing for freedom (freedom as trajectory, not snapshot), voluntary delegation as a distinct mechanism from coercive lock-in, the infrastructure dependency stack below the application layer, contributor obligations as the missing counterpart to user rights, and freedom-across-time as the seventh unresolved tension. These additions expand the design space without resolving the tensions the adversarial pass correctly identified.

## Changes Made

- **Added temporal framing** as a fifth alternative framing under Theoretical Grounding: freedom is a system trajectory, not a point-in-time property; a free-at-release system can become unfree through drift alone.
- **Added Voluntary Delegation section** under User Sovereignty §1: distinguishes coercive lock-in from genuine user preference and structural coercion via network effects; surfaces three interpretations with different design implications.
- **Added contributor obligations note** under Shared Improvement Loop: the rights-focused framing has no counterpart obligations; names upkeep, attribution, and non-weaponization obligations as candidates.
- **Added minimal footprint** to Preferred System Characteristics: each required infrastructure component is a potential freedom-reducing dependency.
- **Added Infrastructure Dependency Stack section** under Architectural Implications: tables the freedom risks at each layer (hardware through application) and proposes preferring commons-governed infrastructure dependencies over commercially controlled ones.
- **Added Tension #7: Freedom Across Time** under Unresolved Tensions: names freedom decay as a distinct failure mode from adversarial capture; raises periodic freedom assessments and reversibility as open questions.
- **Updated document status** from `adversarial` to `strawman` in frontmatter.
- **Added decision record entry** for this pass.

## Why These Changes Matter

The adversarial pass correctly identified that the proposal's failure modes are structural, not incidental. But the response to structural critique is not just annotation — it is expanding the design space to find framings and mechanisms that might survive those critiques. The temporal framing reframes "what is a free system" in a way that might be more resistant to the economic gravity failure mode, because it makes freedom maintenance an ongoing requirement rather than a one-time architectural achievement. The infrastructure dependency stack surfaces a ceiling on application-layer freedom that has no current treatment. The voluntary delegation section opens a genuine philosophical fork that changes what the proposal is trying to accomplish depending on which interpretation is adopted.
